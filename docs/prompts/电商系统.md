# 1. 项目背景与目标

## 1.1 项目背景

随着线上购物方式的普及，电商商城系统已经成为商品展示、用户购买和订单管理的重要平台。为了理解电商系统的基本业务流程，并完成课程作业要求，本项目计划开发一套简易版电商商城系统。

本项目采用前后端分离架构，前端负责页面展示和用户交互，后端负责业务逻辑处理，数据库负责保存用户、商品、购物车和订单等数据。

由于本项目属于课程作业性质，因此系统不追求商业级复杂功能，只实现电商系统中最核心的业务闭环，包括用户注册登录、商品浏览、加入购物车、提交订单、模拟支付、后台商品管理和订单发货等功能。

# 2. 业务场景

## 场景一：用户在线购买商品

用户进入商城首页，浏览商品列表，选择感兴趣的商品查看详情。用户登录后，可以将商品加入购物车，在购物车页面填写收货信息并提交订单。订单生成后，用户可以进行模拟支付，订单状态从“待支付”变为“已支付”。

业务流程：

```text
用户登录 → 浏览商品 → 查看详情 → 加入购物车 → 提交订单 → 模拟支付 → 查看订单
```

业务痛点：

```text
如果没有完整购物流程，系统只能展示商品，不能体现电商业务闭环。
如果没有购物车和订单，项目功能会显得过于简单。
```

---

## 场景二：管理员维护商品信息

管理员登录后台后，可以查看商品列表，并对商品进行新增、修改、删除操作。商品数据更新后，用户端商城首页可以展示最新商品信息。

业务流程：

```text
管理员登录 → 查看商品列表 → 新增商品 / 修改商品 / 删除商品 → 用户端同步展示
```

业务痛点：

```text
如果商品只能写死在前端，系统不具备后台管理能力。
如果商品不能动态维护，项目不符合真实商城系统的基本要求。
```

---

## 场景三：管理员处理订单发货

用户完成模拟支付后，管理员可以在后台订单管理页面查看订单，并对已支付订单进行发货操作。发货后订单状态变为“已发货”。

业务流程：

```text
用户支付订单 → 管理员查看订单 → 管理员点击发货 → 订单状态更新
```

业务痛点：

```text
如果订单提交后没有后台处理流程，商城业务链路不完整。
后台发货功能可以体现管理员端与用户端之间的数据联动。
```

# 4. 详细功能说明

## 4.1 用户注册

### 接口

```text
POST /api/user/register
```

### 请求参数

```json
{
  "username": "user",
  "password": "123456",
  "nickname": "测试用户"
}
```

### 技术要求

```text
1. username 不能为空。
2. password 不能为空。
3. username 不能重复。
4. 注册成功后写入 user 表。
5. 返回统一 Result 格式。
```

### 验收标准

```text
1. 输入新用户名可以注册成功。
2. user 表产生新用户记录。
3. 使用重复用户名注册时返回“用户名已存在”。
```

---

## 4.2 用户登录

### 接口

```text
POST /api/user/login
```

### 请求参数

```json
{
  "username": "user",
  "password": "123456"
}
```

### 技术要求

```text
1. 根据 username 查询用户。
2. 用户不存在时返回错误。
3. 密码错误时返回错误。
4. 登录成功返回 id、username、nickname。
5. 前端保存 userInfo 到 localStorage。
```

### 验收标准

```text
1. user/123456 可以登录。
2. 登录成功后跳转 /home。
3. localStorage 中存在 userInfo。
```

---

## 4.3 商品列表

### 接口

```text
GET /api/product/list
```

### 技术要求

```text
1. 查询 product 表中 status = 1 的商品。
2. 返回商品 id、name、category、price、stock、imageUrl、description、status。
3. 首页使用商品卡片展示。
```

### 验收标准

```text
1. 首页能显示初始化商品。
2. 商品包含图片、名称、分类、价格、库存。
3. 点击“查看详情”可以进入商品详情页。
```

---

## 4.4 商品详情

### 接口

```text
GET /api/product/{id}
```

### 技术要求

```text
1. 根据商品 ID 查询商品。
2. 商品不存在时返回错误。

```

### 验收标准

```text
1. 点击商品后进入 /product/:id。

```

```text
1. 判断用户是否存在。
2. 判断商品是否存在。
3. 判断商品是否上架。
4. 判断库存是否足够。
5. 如果购物车已有同一商品，则数量累加。
6. 如果购物车没有该商品，则新增购物车记录。
7. 累加后数量不能超过库存。
```

### 验收标准

```text
1. 登录用户可以加入购物车。
2. 同一商品重复加入时购物车数量累加。
3. 库存不足时提示错误。
```

## 4.9 创建订单

### 接口

```text
POST /api/order/create
```

### 请求参数

```json
{
  "userId": 1,
  "receiverName": "张三",
  "receiverPhone": "13800000000",
  "receiverAddress": "广东省广州市天河区"
}
```

### 技术要求

```text
1. 判断用户是否存在。
2. 校验收货人、手机号、收货地址不能为空。
3. 查询当前用户购物车。
4. 购物车为空时返回错误。
5. 根据商品 ID 查询真实商品价格和库存。
6. 后端计算订单总金额。
7. 生成订单编号。
8. 写入 orders 表。
9. 写入 order_item 表。
10. 扣减 product 表库存。
11. 清空当前用户 cart 表。
```

### 重要限制

```text
前端不能传订单金额。
后端不能相信前端价格。
订单金额必须以后端查询到的商品价格为准。
```

### 验收标准

```text
1. 提交订单后 orders 表新增订单。
2. order_item 表新增订单明细。
3. 商品库存减少。
4. 当前用户购物车被清空。
5. 页面跳转到我的订单页。
```

# 13. 总结

本项目是一个基于 Spring Boot + Vue 的简易电商商城系统 MVP，主要用于课程作业展示。系统采用前后端分离架构，前端使用 Vue 3 和 Element Plus 实现页面展示，后端使用 Spring Boot 和 MyBatis-Plus 提供接口服务，数据库使用 MySQL 保存业务数据。

项目功能虽然经过简化，但覆盖了电商系统的核心业务流程，包括用户注册登录、商品浏览、购物车、订单提交、模拟支付、后台商品管理和订单发货，能够满足课程作业中对“可运行、可演示、有业务闭环”的基本要求。
